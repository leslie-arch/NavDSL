1. FOL Nav-DSL

Habitat 认知导航可结合的研究热点

  1. 语义场景图 (3D Scene Graphs) 与 Z3 的结合

目前 Habitat 领域的热点是将非结构化的点云转化为分层语义场景图 (Hierarchical 3D Scene Graphs)。结合点： 场景图中的“节点”（房间、家具、物体）和“关系”（在…里面、靠近、连接）可以直接映射为你 DSL 中的谓词 (Predicates)。价值： 你不需要自己写视觉算法去提取 At(robot, kitchen)，你可以利用成熟的场景图算法（如 Hydra 或 ConceptGraphs）作为 Z3 的“事实输入层”。

  1. 具身重排任务 (Rearrangement Tasks)

这是 Habitat 3.0 后的核心挑战：机器人不仅要导航，还要移动物体。结合点： 重排任务涉及复杂的长程因果链（例如：为了把牛奶放进冰箱，必须先放下手中的面包，再打开冰箱门）。价值： 这正是 Z3 的强项。你可以将重排任务中的“状态改变”作为 Z3 的约束，验证 LLM 生成的搬运指令序列是否物理可行。

  1. 人机协作与社交约束 (Social Navigation)

Habitat 3.0 引入了类人智能体。结合点： 在导航中加入社交逻辑约束。例如，Z3 约束：Implies(Moving(robot), Distance(robot, human) > 1.5m)。价值： 这能将你的研究从单纯的“逻辑导航”提升到“安全可靠的社会机器人”高度。

1.1. Nav-DSL 谓词表定义 (Predicate Table)

谓词是描述环境状态和任务约束的基本单元。

A. 状态谓词 (State Predicates) — 描述“是什么”

静态环境谓词 (Static Environment Predicates)用于描述不随动作轻易改变的环境属性。

  • \(Is(o, a)\): 物体 \(o\) 具有属性 \(a\)（如：Is(door1, Locked)）。
  • \(Connected(l_1, l_2, d)\): 位置 \(l_1\) 与 \(l_2\) 通过通道 \(d\) 相连（如：门或走廊）。
  • \(Located(o, l)\): 物体 \(o\) 的初始坐标或区域位于 \(l\)。

动作状态谓词(Dynamic State Predicates)用于描述随机器人动作演变的状态，抽象为时间步索引的谓词集合，是 Z3 检查时序逻辑的核心。

  • \(At(r, l, t)\): 在时间步 \(t\)，机器人 \(r\) 位于位置 \(l\)。 空间谓词
  • \(Holding(r, o, t)\): 在时间步 \(t\)，机器人 \(r\) 持有物体 \(o\)。 交互谓词
  • \(Opened(d, t)\): 在时间步 \(t\)，通道 \(d\) 处于开启状态。 状态谓词
谓词 (Predicate) 语义说明 (Semantics) Z3 类型定义
At(robot, location) 机器人当前处于位置 location Function(Object, Location, Bool)
In(object, container) 物体 object 在容器 container 内 Function(Object, Object, Bool)
Is(object, attribute) 物体具有某种属性（如 hot, red, locked） Function(Object, Attribute, Bool)
Connected(loc1, loc2) 位置 1 和位置 2 之间物理连通 Function(Location, Location, Bool)
Holding(robot, object) 机器人当前抓取着物体 object Function(Object, Bool)

B. 动作算子 (Action Operators) — 描述“做什么”

为了让 Z3 能够进行推理，每个动作由前提条件 (Pre-condition) 和 效应 (Effect) 组成。

动作算子 Act(args) 前提条件 Pre(Act) 效应 Eff(Act)
\(Move(r, l_1, l_2, t)\) \(At(r, l_1, t) \wedge Connected(l_1, l_2, d) \wedge Opened(d, t)\) \(At(r, l_2, t+1) \wedge \neg At(r, l_1, t+1)\)
\(Pick(r, o, l, t)\) \(At(r, l, t) \wedge Located(o, l) \wedge \neg \exists x Holding(r, x, t)\) \(Holding(r, o, t+1)\)
\(Unlock(r, d, k, t)\) \(At(r, l_{door}, t) \wedge Holding(r, k, t) \wedge Is(k, KeyFor\_d)\) \(\neg Is(d, Locked) \wedge Opened(d, t+1)\)
动作 (Action) 参数 前提条件 (Pre-conditions) 效应 (Effects)
Goto (targetloc) Connected(currentloc, targetloc) At(robot, targetloc)
PickUp (obj) At(robot, objloc) ∧ ¬Holding(robot, any) Holding(robot, obj)
Open (obj) At(robot, objloc) ∧ Is(obj, unlockable) Is(obj, opened)
Check (obj, attr) At(robot, objloc) Knowledge(Is(obj, attr))

1.2. DSL 语法结构设计 (Syntax Structure)

为了让 LLM 能够稳定生成，建议采用 S-表达式 (S-Expression) 或 JSON-Logic 风格。这里推荐 类 Python 表达式，因为它对 Z3Py 最友好。

A. 任务序列结构一个完整的任务被解析为三个部分：

Objects: 涉及的实体声明。 InitialState: 环境当前已知的真值。 PlanSteps: LLM 建议的动作链。 Constraints: 必须满足的硬性逻辑约束。

B. 形式化语法示例 (Backus-Naur Form, BNF) 

<Task>        ::= <ObjectDecl> <InitState> <GoalState> <ActionSequence>
<Action>      ::= <ActionName> "(" <ParameterList> ")"
<Constraint>  ::= "Require(" <LogicFormula> ")"
<LogicFormula>::= <Predicate> | "And(" <FormulaList> ")" | "Implies(" <Formula>, <Formula> ")" | "Not(" <Formula> ")"

1.2.1. 基于 LLM 的语义-逻辑翻译机制

DSL 语法结构与翻译流程 需要定义一套 LLM 易于生成且 Z3 易于解析的语法结构（建议采用类似 LISP 表达式 或 Pythonic Predicates）。

[语义 - 逻辑翻译架构图]

  1. 2.1 引导式 Prompt 工程

    设计了一种带约束的 Prompt 架构，包含：

    谓词字典约束：强制 LLM 仅能使用预定义的谓词和动作算子。上下文示例（Few-shot）：展示如何处理隐含的逻辑，例如指令“如果门关了就找钥匙”需翻译为条件逻辑分支 If(Locked(d), Then(Search(k)))。

  2. 2.2 语法良构性检查

    生成的 DSL 代码通过一个轻量级的 Python 解析器，检查参数类型是否匹配（例如 Move 的参数必须是 Location 类型），确保进入 SMT 求解器的输入具有语法正确性。

    A. 语义解析层 (Semantic Parsing Layer) LLM 接收自然语言指令 \(I\)，输出 DSL 序列 $S$：

    • 输入： “去办公室拿蓝色文件夹，如果门锁了，去前台取钥匙。”

    • DSL 输出: 

      # Logic Objects Declaration
Objects(robot, office, blue_folder, desk, key, gate)

# Sequence of Hypothetical Actions
Step1: Move(robot, hallway, office, gate)
Step2: If(Is(gate, Locked), 
          Then=[Move(robot, hallway, reception), Pick(robot, key), Move(robot, reception, office)],
          Else=[Continue])
Step3: Pick(robot, blue_folder, office)

    B. 语法规范约束 (Syntactic Constraints) 为了保证 LLM 不乱写，你需要在方法论中定义 Well-formedness (良构性)规则：

    • 类型检查： 谓词的参数必须属于已定义的 Objects 集合。
    • 时序连续性： \(At(r, l, t+1)\) 的前提必须在 \(t\) 时刻有合法的移动动作。
    • 单手约束： 任何时刻 \(t\)，满足 \(Holding(r, o, t)\) 的 \(o\) 数量 \(\le 1\)。
  3. 基于 Z3 的形式化验证与真值推演

    系统的“理智大脑（System 2）”，负责对翻译出的逻辑序列进行物理一致性验证。

    3.1 逻辑可满足性建模系统将任务目标（Goal）与当前已知环境状态（Init）编码为 SMT 问题： \[\Phi = \text{Init} \wedge \text{Constraints} \wedge \text{Action\_Sequence} \wedge \neg \text{Goal}\] 利用 Z3 求解器搜索是否存在一个合法的状态演化路径。

  4. 融合因果图的动态重规划机制

    在复杂动态环境中，机器人通过感知更新实时修正其因果知识图谱。

    • 4.1 因果关联建模

    利用因果发现算法维护环境实体间的依赖关系（如：电源开关与灯光亮度、门锁状态与路径连通性）。

    • 4.2 反事实重规划

    当执行过程中遭遇意外（如路径被突发障碍堵塞），系统利用因果图进行反事实推理：“如果不走 A 门，是否可以绕过 B 窗？”。 Z3 重新计算最优备选逻辑链，确保导航任务在动态环境下的稳健性。

1.3. 语义-逻辑翻译示例 (Semantic-to-Logic Mapping)

论文实验部分的核心：展示从“人话”到“DSL”的转化。

自然语言指令： “去厨房拿那个热杯子，如果门锁了就先找钥匙。” Nav-DSL 映射结果： 

# 1. 对象声明
Objects = [Robot(r), Room(kitchen), Item(cup), Item(key), Door(d)]

# 2. 初始知识 (Initial Knowledge)
Init = [
    At(r, living_room),
    Connected(living_room, kitchen, via=d),
    Is(cup, hot)
]

# 3. 逻辑约束 (Constraints / Pre-conditions)
# 这一部分是 Z3 的核心，由系统根据 DSL 库自动补全
Constraints = [
    # 如果门锁了，进入厨房的前提是必须拿着钥匙
    Implies(And(Is(d, locked), At(r, kitchen)), Holding(r, key)),
    # 拿到杯子的前提是必须在厨房且杯子是热的
    Implies(Holding(r, cup), And(At(r, kitchen), Is(cup, hot)))
]

# 4. LLM 生成的候选计划 (Candidate Plan)
Plan = [
    Goto(kitchen),  # 此处若门锁且无钥匙，Z3 将返回 UNSAT
    PickUp(cup)
]

1.4. 实验数据

1.4.1. 实验指标定义 (Metrics)

  1. SR (Success Rate): 解析出的 DSL 语法完全正确的比例。
  2. LV (Logical Validity): 经 Z3 验证，计划在物理逻辑上可达的比例。
  3. CR (Correction Rate): 经过 Z3 反馈给 LLM 后，第二次修正成功的比例。

1.4.2. 实验方案 (Experimental Protocol)

结合 2025-2026 年具身智能顶会（CVPR, ICRA, ICLR）的趋势，建议采用以下方案：

  1. 环境与基准设置 (Simulation Setup)
    • 仿真器： 使用 Habitat 3.0+。重点利用其支持动态物体交互（如门、抽屉、可移动物体）的特性。
    • 场景库： 选取 HM3D 或 HSSD 中的 50 个复杂户型，确保包含多楼层和封闭空间。

    • 数据集构建： 不要只用模板生成的指令。利用 GPT-4 辅助生成 200 条具有“逻辑诡计”的指令，例如 

      "Go fetch the cold soda from the fridge, but don't walk through the living room because the floor is wet." (测试路径约束 + 状态约束)
  2. 评估指标 (Evaluation Metrics) 除了常规的成功率，增加以下硬指标：
    • SR (Success Rate): 任务最终完成的比例。
    • SPL (Success weighted by Path Length): 导航效率。
    • \(\text{SR}_{formal}\) (Formal Validity Rate): 生成的计划在执行前通过逻辑检查的比例。
    • Correction Gain: 引入纠错环节后，成功率提升的绝对值。
  3. 消融实验 (Ablation Study) 必须包含以下变体以证明 Nav-DSL 的必要性：
    • w/o Nav-DSL: LLM 直接生成 PDDL 或 Python 代码。
    • w/o Z3 Verification: LLM 生成 DSL 后直接执行，不经过求解器。
    • w/o Unsat Feedback: 失败后只告诉 LLM "Wrong"，不提供冲突点。
  4. 案例研究 (Case Study Visualization) 在论文中留出半页纸，做一个“逻辑对比图”。展示一个任务：“拿钥匙开门去阳台”。
    • Baseline： 机器人直接撞门（因为 LLM 忘了开门）。
    • Ours： 展示 Z3 抛出 Unsat: Requires(At(r, balcony), Open(door)) \(\rightarrow\) 反馈给 LLM \(\rightarrow\) LLM 修正为 [Pick(key), Unlock(door), Move(balcony)]。

1.4.3. Nav-DSL 实验数据集模板-—语义-逻辑映射对照表

  • 难度 1：基础线性任务 (Basic Linear Tasks)
    • 自然语言指令 (NL)： "Go to the kitchen and fetch the blue mug."

    • 语义逻辑解析 (Nav-DSL)： 

      Objects(robot, kitchen, blue_mug, cupboard)

Plan: [
    Move(robot, current_loc, kitchen, door_1),

      Objects(robot, office, glass_door, reception, key)
# 核心逻辑：条件分支翻译
If(Property(glass_door, Locked, t1),
   Then=[Move(robot, hallway, reception), Pick(robot, key), Move(robot, reception, office)],
   Else=[Move(robot, hallway, office)])
  • 难度 3：含逻辑冲突的陷阱指令 (Conflict Tasks - 用于测试 Z3)
  • 自然语言指令 (NL)： "Directly enter the storage room and pick up the heavy box, then take the small box immediately."

  • 语义逻辑解析 (Nav-DSL)： 

    Objects(robot, storage, heavy_box, small_box)
  Plan: [
    Move(robot, hallway, storage, door_2),
    Pick(robot, heavy_box, storage),
    Pick(robot, small_box, storage) # 此处将触发 Z3 冲突：Single-hand Constraint
]

1.4.4. System Prompt

采用了**“定义-规范-示例”**的结构，能够极大地降低 LLM 输出的随机性，确保生成的 DSL 逻辑表达式可以直接对接 Z3 求解器。 

你是一名精通符号逻辑与机器人任务规划的“神经符号转换专家”。你的任务是将用户的自然语言指令（Natural Language）精准地翻译为
我定义的 Nav-DSL 逻辑表达式。这些表达式随后将被输入 Z3 SMT 求解器进行形式化验证。
1. Nav-DSL 谓词库 (Predicate Library):
   At(robot, location, time): 机器人在 $t$ 时刻处于某位置。
   Holding(robot, object, time): 机器人在 $t$ 时刻持有某物体。
   Connected(loc1, loc2): 两个位置在物理上连通。
   Is(object, attribute, time): 物体在 $t$ 时刻具有某种属性（如：Locked, Open, Hot）。
2. 动作算子规范 (Action Schemas):
   Move(robot, from_loc, to_loc, time): 移动。
   Pick(robot, object, loc, time): 抓取（前提：机器人必须在该位置且手部为空）。
   Unlock(robot, door, key, time): 解锁（前提：机器人持有钥匙且在门前）。
   Open(robot, container, time): 打开。
3. 约束规则 (Hard Constraints):
   单手约束：同一时刻机器人只能 Holding 一个物体。
   时序连续性：时间步必须从 $t=0$ 开始，连续递增（$t_0, t_1, t_2 \dots$）。
   逻辑分支：使用 If(Condition, Then=[], Else=[]) 处理不确定性指令。
4. 输出格式要求:请严格按以下结构输出，不要包含任何解释性文字：
   #+BEGIN_SRC python
        # Objects: [实体列表]
        # Init: [初始状态谓词]
        # Goal: [目标状态谓词]
        # Plan: [按时间步排列的动作算子序列]
   #+END_SRC
5. 示例 (Few-shot Examples):
   - 输入: "Go to the kitchen and grab the coffee mug."
   - 输出:
     #+BEGIN_SRC python
     Objects: [robot, hallway, kitchen, coffee_mug]
     Init: [At(robot, hallway, 0), Connected(hallway, kitchen)]
     Goal: [Holding(robot, coffee_mug, t_final)]
     Plan: [
         Move(robot, hallway, kitchen, 0),
         Pick(robot, coffee_mug, kitchen, 1)
     ]
     #+END_SRC
  1. 快速生成论文所需的 Table 1 (LLM Translation Accuracy)：
    1. 第一步：准备测试集 (Input Sentences)

    准备 20-30 个包含不同逻辑挑战的句子，例如：简单: "Go to the office." 含隐含动作: "Bring me the hot water from the pantry." (隐含了：移动、抓取、移动回来) 含逻辑陷阱: "Pick up the laptop and the apple at the same time." (用于测试 Z3 是否能拦截单手约束) 含条件分支: "Go to the lab; if the door is shut, find the key card."

    1. 第二步：运行翻译并记录结果

    将上述句子喂给 LLM。你会发现即便 LLM 翻译出了代码，Z3 在验证时会报出 unsat。论文亮点点拨：你可以写道：“实验发现，即便最先进的 LLM（如 GPT-4）在面对‘隐含约束’时仍会生成违反物理规律的代码，而本文提出的 Nav-DSL 配合 Z3 验证能有效拦截 100% 的此类逻辑错误。”

    1. 第三步：数据整理建议表格

1.4.5. LLM 逻辑纠错反馈 Prompt (Self-Correction Prompt)

在神经符号架构中，当 Z3 求解器返回 unsat 时，最关键的步骤是将底层逻辑冲突（Unsat Core）转化为高层语义建议。 

Role:
你是一名具身智能规划调试专家。当前的机器人任务计划由于违反了物理常识或逻辑约束，未通过形式化验证（Z3 Solver）。
请根据反馈的错误原因，重新优化你的逻辑表达式（Nav-DSL）。

1. 上下文信息 (Context):
原始指令 (Original Instruction): {user_instruction}
失败的计划 (Failed DSL Plan): ```python
{failed_dsl_plan}
```
2. 逻辑冲突反馈 (Z3 Feedback):
> 检测到逻辑矛盾: {logical_violation_description}
> 例如：动作 `Pick(box2)` 失败，因为在 $t=2$ 时刻机器人已持有了 `box1`，违反了单手约束。

3. 修正要求 (Requirements):
- **定位冲突**：分析报错中提到的谓词和时间步。
- **补全动作**：如果是因为缺少前提条件（如没拿钥匙、没开门），请在合适的时间步插入缺失动作。
- **调整顺序**：如果是因为动作顺序错误，请重新排布时间步。
- **输出规范**：依然严格遵守 Nav-DSL 格式（Objects, Init, Goal, Plan），不要包含任何解释性文字。

**4. 修正后的输出 (Corrected DSL Output):**
*(请开始生成修正后的代码)*

1.5. 创新性：

确定性映射 (Deterministic Mapping)： 强调你的 DSL 消除了解析中的歧义。例如，“热杯子”被精准映射为 Is(cup, hot)，而不是模糊的视觉特征。因果先验嵌入 (Causal Prior Embedding)： 在 DSL 动作算子中，你预设了物理逻辑（如：只有 Holding 才能 Drop）。这使得 Z3 能够对 LLM 生成的“反物理”计划进行一票否决。双向翻译机制 (Bi-directional Loop)：

  • 正向： 自然语言 -> LLM -> Nav-DSL -> Z3。
  • 逆向： Z3 (Unsat Core) -> 错误解释器 -> 自然语言反馈 -> LLM 修正。

本文方法处于 RL 和 Ageng Skills 两种范式的上层逻辑控制位：

与 Agent Skills 的联系
Nav-DSL 动作算子（如 Pick(obj)）实际上就是 Agent Skills 的“语义接口”。LLM 生成逻辑计划，Z3 验证计划，而最终的执行则是由底层的 Agent Skills（如平滑的抓取控制器）完成。
与 RL 的联系
RL 通常用于训练底层的感知或运动控制（如避障）。本文方法为 RL 提供了一个**“逻辑护栏（Safety Guardrail）”**，即通过 Z3 过滤掉逻辑错误的指令，只让 RL 去执行逻辑上可行的路径。

相比于 RL 的“暴力训练”和 Agent Skills 的“概率盲盒”，Nav-DSL + Z3 的优势体现在以下三个方面：

A. 可证明的安全性与可行性 (Provable Feasibility)

  • 对比： RL 可能在导航中撞墙千次才学会绕行；LLM 可能会建议机器人“穿过关着的门”。
  • 优势： 在指令下发给机器人之前，就已经通过 Z3 证明了该动作序列在物理拓扑和前提条件下是**可满足（Satisfiable）**的。这种“先验证再执行”的机制是具身智能走向工业级应用的关键。

B. 零样本的长程逻辑处理能力 (Zero-shot Long-horizon Planning)

  • 对比： 任务链越长，RL 的奖励越稀疏，训练越难收敛；Agent Skills 的 LLM 容易在第 5 步之后忘记第 1 步的约束。
  • 优势： SMT 求解器处理 10 步甚至 100 步的逻辑链条在计算上是廉价且严谨的。Nav-DSL 能精准捕捉“如果要完成 A，必须先完成 B”这种深层依赖。

C. 基于真值的解释性纠错 (Explainability via Unsat Core)

  • 对比： RL 失败了只会返回一个低分；LLM 失败了可能会一本正经地胡说八道。
  • 优势： 当 Z3 返回 unsat 时，它能通过 Unsat Core 明确指出：“冲突点在于：你试图打开门 D1，但你并未持有钥匙 K1”。这种基于逻辑事实的反馈比任何统计模型的猜测都更具说服力，也让系统的自纠错变得极度高效。
  1. Comparison with Existing Paradigms (RL & Skill-based Agents)

    While current embodied AI research primarily relies on Reinforcement Learning (RL) and Skill-based LLM Agents, our proposed Nav-DSL + SMT framework introduces a fundamental shift from stochastic association to deterministic logical deduction. Firstly, unlike Reinforcement Learning, which seeks optimal policies through extensive empirical trial-and-error, our method eliminates the need for dense reward engineering and massive sampling. While RL excels in low-level reactive control, it lacks the formal guarantees required for high-level task consistency. In contrast, by grounding semantic instructions into SMT constraints, our framework ensures that the generated task sequences are provably feasible within the defined physical topology before any execution occurs. Secondly, compared to Skill-based Agents that sequence behavioral primitives via LLM-based prompting, our approach addresses the critical issue of "logical hallucinations." Even the most advanced LLMs often overlook implicit pre-conditions (e.g., trying to open a locked door without a key) due to their probabilistic nature. Nav-DSL serves as a formal bridge, transforming fluid neural semantics into rigorous symbolic predicates. The integration of the Z3 SMT solver allows the system to detect unsatisfiable plans in milliseconds. Distinctively, our framework provides explainable failure recovery through Unsat Core tracing. Unlike RL agents that return a low scalar reward upon failure, or LLM agents that provide vague linguistic guesses, our system pinpoints the exact logical contradiction (e.g., a violation of the single-hand constraint). This feedback loop enables a closed-loop refinement of task plans, significantly enhancing the reliability and transparency of cognitive navigation in complex, dynamic environments.

  2. Symbolic Verification vs. Parameter Scaling

    在 2026 年的学术语境下，单纯堆砌模型参数（Scaling Law）已进入边际效用递减期，而**“可验证的具身推理”**正是当前的破局点。

    A Strategic DiscussionA prevailing trend in Embodied AI is the pursuit of "Emergent Reasoning" through the continuous scaling of Large Language Model (LLM) parameters. However, our study argues that for cognitive navigation in safety-critical environments, incorporating a formal SMT solver (like Z3) is fundamentally more robust and efficient than simply increasing model scale.

    1. The Fallacy of Probabilistic Reliability :: > Even as LLMs scale to trillions of parameters, they remain stochastic simulators of linguistic patterns. In embodied tasks, a plan that is "\(99\%\) linguistically plausible" can be "\(100\%\) physically catastrophic$." For instance, an LLM might generate a visually coherent sequence that violates a subtle temporal constraint, such as \(\neg Holding(r, key, t) \Rightarrow \neg Open(r, door, t+1)\). While scaling may reduce the frequency of such hallucinations, it cannot provide a formal guarantee of their absence. By contrast, our Nav-DSL + Z3 framework treats physical laws as invariants (\(\phi_{inv}\)), ensuring that any plan bypassing the verification layer is logically sound by construction.
    2. Computational Efficiency and Edge-Case Handling :: > Scaling laws entail massive computational overhead and energy consumption, yet LLMs still struggle with "long-tail" logical edge cases. Our approach demonstrates a computational decoupling: we utilize a relatively lightweight LLM for high-level semantic "compilation" and delegate the heavy lifting of combinatorial logic to the SMT solver. The solver’s ability to pinpoint the Unsat Core provides a level of precision in failure diagnosis that neural scaling cannot replicate. Instead of "brute-forcing" logic through parameters, we utilize Symbolic Anchoring to prune the search space, achieving higher reliability with significantly lower inference costs.
    3. Neuro-Symbolic Synergy in Agentic Workflows :: > By 2026, the transition toward "Agentic" robot frameworks necessitates a shift from black-box execution to interpretable orchestration. Our framework positions the LLM as the "Intuitive Brain" and the SMT solver as the "Logical Auditor." This synergy ensures that the agent's autonomy is bounded by a verified safety envelope, a prerequisite for deploying autonomous systems in unpredictable human-centric environments.
